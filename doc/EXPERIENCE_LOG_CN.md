# 项目经验教训志：神经符号物理求解器

## 1. 技术“战壕”：长推理模型的现实挑战

在使用 **DeepSeek-R1**（长推理模型）进行开发时，我们遇到了标准 LLM 处理流程中不常见的独特挑战。

### 网络不稳定性与 KeyboardInterrupt
- **问题**：R1 在输出最终 JSON 之前，通常会在“思考”块中花费 60-120 秒。在此窗口期间，任何网络波动或超市都会触发 Python 客户端的 `KeyboardInterrupt`。
- **教训**：我们必须实现**持久化树日志** (`tree_log.json`)。这允许协调器从中断处精确恢复，将潜在的失败转化为“暂停/恢复”工作流。

### API JSON 提取的博弈
- **问题**：早期版本的 `TheoristAgent` 使用惰性正则 `(.*?)` 提取 JSON，偶尔会截断较长的数学建议。
- **修复**：切换到贪婪正则 `(.*)` 并增加了寻找最外层 `[` 和 `]` 括号的后备逻辑。

## 2. 策略转变：从朴素 DFS 到优先级 BFS

最初，求解器采用深度优先搜索，这容易陷入“数学死胡同”（例如，对简单的积分尝试无穷级数展开）。

### 转向最佳优先搜索 (BFS)
- **决策**：我们使用 `heapq` 实现了优先级队列。
- **启发式算法**：$优先级 = 成功概率 \times 0.9^{深度}$。
- **平局处理**：在优先级元组中增加了计数器，以处理两个路径概率相同的情况，防止堆操作时出现 `TypeError`。

## 3. 精度与验证逻辑的突破
最关键的 Bug 是验证过程中的“推导不匹配”。

### 微积分感知验证 (Calculus-Aware Verification)
- **问题**：系统最初只检查直接等价性。然而，“积分”步骤意味着*父节点*是*子节点*的导数。
- **修复**：我们修改了 `orchestrator.py` 以执行“多重匹配检查”：
    - 匹配 1：$父节点 \approx 子节点$
    - 匹配 2：$父节点 \approx \frac{d}{da}子节点$（积分步）
    - 匹配 3：$\frac{d}{da}父节点 \approx 子节点$（微分步）

## 4. 关键心得
- **被积函数隔离**：在清理 `Eq()` 对象的同时保留其结构，对于 $I(a)$ vs $I'(a)$ 的逻辑至关重要。
- **Oracle 鲁棒性**：`NumericalOracle` 必须足够模块化，以处理递归积分和像 $C$ 这样的符号常数。
- **日志即架构**：`thinking_process.txt` 不仅仅是日志；它是系统外部化的“工作记忆”，允许用户在不堵塞终端的情况下审计 AI 的内部推导。
